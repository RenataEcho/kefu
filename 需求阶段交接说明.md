## 二、本阶段已完成的工作

### 1. 需求补全访谈
基于原始流程文档 `客户服务流转中心.md`，进行了需求访谈，整理了完整需求补全记录。

### 2. Edge Case Hunter 扫描（第一轮）
发现 13 个未处理边界场景，全部经业务方确认后落入文档，包括：
- Excel 批量导入重复编码处理规则
- 已付费用户重复录入拦截
- SLA 自然日定义 + 二次催促规则
- 微信服务号通知失败降级短信
- 审核被拒后重新提交的记录归档逻辑
- API 不可用时各模块降级行为
- 右豹ID 与右豹编码为两个不同字段
- 等共 22 条已确认项

### 3. Adversarial Review（对抗性审查）
对两份文档进行联合审查，发现 14 个问题，全部经业务方逐条答复并落入文档，主要补充内容：
- 付费记录支持修改（金额/时间/对接人）和删除（含状态回退）
- 非付费用户流程重写：客服直接在系统录入，飞书API获取入群申请
- 新增"入群审核表"完整数据结构（12个字段）
- 新增"模块6：权限控制"（菜单/操作/字段三维度 + 操作日志）
- 通知对象明确为审核员，短信格式含群名称+H5链接
- 旧审核记录归档机制

### 4. 新增三个管理模块
业务方补充了客服管理、导师管理、门派社群管理的需求：
- 新增客服表、导师表、门派表数据结构
- 新增模块7（客服管理）、模块8（导师管理）、模块9（门派社群管理）
- 用户主档表新增所属客服、所属门派、所属导师三个聚合字段
- 导师与学员绑定关系来自第三方系统 API，非本系统录入

### 推荐顺序（在新对话窗口中执行）

**Step 1（可选但推荐）：** 针对新增的三个模块（客服管理/导师管理/门派社群管理）再跑一次 Edge Case Hunter

```
/bmad-review-edge-case-hunter

```

**Step 2（必须）：** 撰写 PRD

当前 BMad 仅安装了 Core 模块，无 PRD 专属技能。可选：
- 直接让 AI 根据 `需求补全记录.md` 生成 PRD 草稿
- 安装 BMad 产品模块后使用结构化 PRD 技能

PRD 需重点补充的内容：
1. 权限角色矩阵（各角色 × 各模块的权限设计）
2. 飞书API 不可用时的降级方案
3. 第三方系统 API 不可用时的降级方案
4. 系统整体页面/菜单导航结构
