# 产品需求文档 · 客户服务中心

**作者：** Maziluo
**日期：** 2026-04-02

## 执行摘要

客户服务中心是为右豹平台内部运营团队专属构建的客服管理后台，将当前分散于企业微信、飞书、Excel 的工作流整合为统一的数字化系统。系统覆盖用户数据录入与校验、入群审核与 SLA 管控、消息通知、飞书好友管理、组织管理及全局运营看板六大核心能力。

**服务对象：** 客服人员（普通客服 / 专属顾问 / 审核员）、导师、系统管理员三类角色，面向右豹平台客服团队全体成员。

**解决的根本问题：** 多工具碎片化操作导致的数据录入错误、流程响应滞后，以及管理层对运营状态的信息断层——随着团队规模持续扩张，这三类问题已从效率损耗升级为业务瓶颈。

**立项背景：** 管理层基础设施主动规划与业务规模爆发的双重驱动，时间窗口明确。

**系统上线后管理层期望首先看到：** 可量化的效率指标（客服处理时长、SLA 达标率）+ 全局运营透明度（实时 Dashboard 替代人工汇报）。

### 核心差异化

**深度定制，非通用 CRM：** 每个字段、每条业务规则、每个操作流程均来自真实业务场景；无冗余功能，无通用 SaaS 适配妥协。

**四路外部 API 原生集成，含完整降级策略：**

| API | 用途 | 降级行为 |
|---|---|---|
| 右豹APP API | 编码校验 + 动作数据同步 | 编码校验失败则阻断提交；动作数据展示缓存 + 提示 |
| 飞书 API | 入群申请获取 + 审核操作 | 管理员手动导入申请数据，SLA 以手动录入时间计算 |
| 第三方导师系统 | 导师学员绑定 + 项目数据 | 展示缓存数据 + 提示暂未更新 |
| 微信服务号 | 审核结果推送 | 推送失败进入失败队列，客服线下跟进 |

**合规可追溯：** 6 张业务表（用户主档、付费记录、入群审核、客服、导师、门派）各自维护独立操作日志，记录操作人、操作时间、操作前后数据快照。

**数据驱动基础设施：** 系统把原本无法度量的人工操作转化为结构化数据，为团队从执行驱动走向数据驱动运营奠定基础。

**不包含：**

- **智能工单分配：** 基于客服负载和用户特征自动分配工单，减少人工调度

- **用户生命周期追踪：** 从首次咨询到付费、晋级、流失的全程数据追踪与干预提示

### 愿景（未来演进）

- **运营数据中台：** 以转化漏斗基线为起点，逐步扩展为支持多维运营决策的数据分析平台
- **自动化工作流引擎：** 基于用户状态变化触发自动化动作（如付费后自动开通学员权益）

## 用户旅程

### 旅程一：小陈的第一天——普通客服完整接入新用户（成功路径）

**角色：** 小陈，右豹平台普通客服，入职3个月，每天处理10-15个新用户接入。

**开场：** 早上9点，小陈打开系统，看到企微上有3个用户昨晚扫码添加了她。其中一个叫刘先生，备注说想了解课程。她需要判断刘先生的意向，并走完接入流程。

**过程：**
1. 小陈判断刘先生暂无付费意向，进入普通用户录入流程
2. 她在系统「用户主档表」新建记录，填入刘先生提供的右豹编码——系统即时调用右豹APP API校验，绿色提示"编码有效"，她松了口气（之前用Excel经常填错）
3. 填入飞书ID、手机号（系统自动校验手机号格式）、飞书昵称，保存
4. 小陈向刘先生发送对应飞书群的二维码，告知申请入群时备注"右豹编码+飞书注册手机号"
5. 两小时后，系统「入群审核工作台」自动出现刘先生的申请记录，审核员小王直接在列表页点击"通过"
6. 系统自动批准飞书入群、推送微信服务号通知——刘先生收到"恭喜您，入群审核已通过"
7. 小陈在Dashboard看到今日建联数+1

**转折（边缘情况）：** 下午小陈录入另一个用户时，右豹编码校验失败——系统提示"编码不存在，请核实后重新录入"。她联系用户重新确认，避免了一次错误数据入库。

**结局：** 小陈完成5个用户录入，没有一次在工具之间切换，没有一条数据因格式错误被退回。

**揭示的能力需求：** 用户主档录入、右豹编码API实时校验、飞书手机号格式校验、入群审核工作台、微信服务号推送、Dashboard个人数据

### 旅程二：小王的紧绷星期三——审核员面对 SLA 超时危机（边缘场景）

**角色：** 小王，右豹平台审核员，兼任专属顾问，同时跟进20+用户，审核工作容易被其他事情挤掉。

**开场：** 周三上午，小王突然收到微信服务号推送："⚠️ 加急提醒：您有3条入群申请已超过3自然日未处理，请立即跟进。"

**过程：**
1. 小王进入「入群审核工作台」——3条记录以红色高亮显示（已超时），旁边还有7条橙色（待审核，未超时）
2. 她依次查看红色记录的详情：右豹编码、飞书昵称、近10天动作数据（关键词申请数、订单数）
3. 第一条：数据正常，一键通过，系统自动批准入群并推送通知
4. 第二条：申请原因疑似异常，小王点击"拒绝"，弹窗填写原因——系统自动推送含原因和H5修改指引的通知
5. 第三条：用户未完成OpenID绑定，推送失败自动进入「通知失败记录」，小王记下右豹编码，下班前通过企微线下联系
6. 处理完毕，超时记录清零，今日SLA压力解除

**8小时后：** 若小王没及时处理，系统会再发一次二次催促——之后不再重复打扰。

**结局：** 小王没有因为遗漏审核被问责，也没有在飞书群里手动翻找申请记录。

**揭示的能力需求：** SLA超时预警（首次+二次）、审核工作台红色/橙色状态标记、拒绝原因弹窗、通知失败队列、飞书入群状态同步

### 旅程四：飞书宕机那个下午——降级操作下的系统韧性（运维场景）

**角色：** 张管理员，系统管理员，负责处理异常情况。

**开场：** 某天下午飞书服务出现故障，系统无法自动拉取新的入群申请。客服群里开始问"为什么审核工作台没有新申请了？"

**过程：**
1. 张管理员收到系统页面提示："飞书API暂时不可用，入群申请自动同步已中断"
2. 他在后台进入手动导入入口，将飞书群管理员手动导出的申请数据以Excel形式上传
3. 系统解析导入数据，生成入群审核记录，SLA计时从导入记录的申请时间字段开始
4. 审核员继续在审核工作台正常操作——通过/拒绝流程不受影响
5. 飞书恢复后，系统自动切回API同步模式，张管理员无需额外操作

**结局：** 飞书故障期间业务未中断，所有申请均有据可查。

**揭示的能力需求：** 飞书API降级手动导入入口、降级状态页面提示、SLA计时以手动录入申请时间为准

## 领域特定需求

本系统为内部运营工具，无行业监管合规要求。以下为影响设计的关键技术约束：

### 数据安全与访问控制

- 所有后台操作需登录认证，会话管理由系统统一控制
- 敏感字段（付费金额、付费对接人、微信OpenID）按角色权限控制显隐
- 微信OpenID仅存储于数据库，不在任何后台管理界面展示
- 操作日志不可删除，作为内部审计的唯一数据源

### 外部集成约束

- 右豹APP API：编码校验为强依赖，不可用时阻断录入提交；动作数据为弱依赖，可展示缓存
- 飞书 API：入群审核为弱依赖，不可用时有人工导入降级路径
- 第三方导师系统 API：弱依赖，不可用时展示缓存数据；返回陌生导师时报错，不自动创建记录
- 微信服务号：通知为弱依赖，失败进队列；无自动短信降级，短信人工执行
- 所有外部 API 调用均通过后端服务中转，前端不直接调用外部接口

### 数据一致性约束

- 右豹编码为全局唯一主键，跨所有业务表的关联基准，录入时强制API校验
- 付费状态变更（付费→未付费）只能由付费记录的删除操作触发，不允许手动直接修改
- 导师门派关系单向维护（导师管理侧），门派侧只读聚合

### 数据同步策略

- 右豹APP动作数据：每小时定时同步 + 支持手动触发；首次录入时异步拉取，期间展示"同步中"
- 飞书好友状态：每天定时更新（非实时），API不可用时展示旧数据+页面顶部提示
- 第三方API数据：Redis缓存，每小时定时刷新，支持手动触发更新

> 相关性能目标（响应时间、可用性要求）详见「非功能需求」章节。

## 管理后台特定需求

### 租户模型

本系统为**单租户内部工具**，仅服务右豹平台一个组织。无多租户隔离需求，无订阅分层，无跨组织数据隔离要求。

### 权限与账号管理（RBAC 矩阵）

#### 权限控制三维度

| 维度 | 说明 |
|---|---|
| 菜单权限 | 控制角色可见/可访问的菜单项 |
| 操作权限 | 控制同一菜单下的新增/修改/删除/导入/导出操作 |
| 字段显隐 | 控制敏感字段（如付费金额、付费对接人）的显示与隐藏 |

#### 审核员说明

"审核员"为管理员账号上的**布尔属性标记**（`is_auditor`），非独立角色。所有被标记为审核员的账号均会收到入群审核相关通知（SLA超时预警、审核驳回通知）。一个账号可同时属于某角色组并被标记为审核员。

#### 建议默认角色组配置

> 以下为系统预设建议，管理员可在系统设置中自定义角色组及其权限。

**角色组一：超级管理员**

| 菜单 | 可见 | 新增 | 修改 | 删除 | 导入 | 导出 | 敏感字段 |
|---|---|---|---|---|---|---|---|
| Dashboard（全局视图） | ✅ | — | — | — | — | — | ✅ |
| 用户主档表 | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ |

## 项目范围界定与分阶段规划

### MVP 策略与理念

**MVP 类型：** 最小可替换系统（Minimum Replaceable System）

**策略依据：** 本系统替代的是企业微信+飞书+Excel的完整工具组合。任何单模块上线均无法形成真实替换——用户仍需维持旧工具，系统价值为零。因此，MVP 的边界不是功能最小化，而是**覆盖全部核心工作流，使团队可以完全切换到新系统**。

**资源需求：**
- 后端：1名 Node.js 或 Python/FastAPI 工程师（负责API中转、SLA定时任务、消息队列）
- 前端：1名 Vue3 工程师（负责管理后台，含复杂页面定制开发）
- 全栈或运维：MySQL + Redis 部署、外部API对接联调

### MVP 核心功能集

MVP 模块清单与范围边界详见「产品范围」章节。以下为各模块的战略必要性说明：

| 模块 | 必要性说明 |
|---|---|
| 用户主档表 + 右豹编码校验 | 核心数据入口，无此无法开始任何流程 |
| 入群审核工作台 + SLA预警 | 直接替代飞书群手动审核，是主要效率提升点 |
| 微信服务号通知（含失败队列） | 审核结果告知用户的唯一渠道 |
| 飞书好友优先级管理 | 替代人工判断添加优先级的核心工具 |
| 客服/导师/门派管理 | 组织数据基础，其他模块依赖 |
| RBAC 权限控制 | 多角色系统的基本安全保障 |
| Dashboard（管理员视图） | 管理层上线首日核心交付物 |
| 历史数据迁移 | 无迁移则系统无法承接现有业务 |

## 功能需求

> 本章节为系统能力契约。所有 FR 描述"系统能做什么"，不涉及实现方式。后续 UX 设计、架构设计、Epic 拆解均以此为边界。

### 用户数据管理

- **FR1：** 客服可以录入用户主档信息（右豹编码、右豹ID、飞书ID、飞书手机号、飞书昵称）
- **FR2：** 系统可以在录入时通过右豹APP API实时校验右豹编码的真实性，校验失败时阻断提交。API不可达（超时或服务不可用）时，普通客服无法绕过，系统提示等待重试；管理员可见「暂跳过校验」按钮，跳过录入的记录必须标记为 `编码待验证` 状态，不得以已验证状态直接入库，API恢复后系统自动在30分钟内触发补校验流程。
- **FR3：** 系统可以自动校验飞书手机号格式，格式不合规时提示错误
- **FR4：** 客服可以查看用户主档列表，包含聚合展示的所属客服、所属导师、所属门派信息
- **FR5：** 客服可以修改用户主档中的可编辑字段
- **FR6：** 管理员可以删除用户主档记录（关联数据>0时系统拦截并提示）
- **FR7：** 负责人可以单条手动新增付费记录
- **FR8：** 负责人可以通过Excel批量导入付费记录，系统识别并报告失败条目，支持导出失败明细
- **FR9：** 系统可以在付费录入时校验合法性：右豹编码须存在、不可重复录入已付费用户、付费金额不可为负数
- **FR10：** 负责人可以修改付费记录的付费金额、付费时间、付费对接人字段
- **FR11：** 负责人可以删除付费记录，删除后对应用户"是否付费学员"状态自动回退为未付费
- **FR12：** 系统可以通过右豹APP API定时同步用户的右豹总关键词数、总订单数、项目收益数据
- **FR13：** 用户可以通过微信服务号H5菜单完成OpenID与右豹编码的绑定

### 入群审核

- **FR14：** 系统可以通过飞书API自动获取飞书群入群申请并生成审核记录
- **FR15：** 管理员可以手动导入入群申请数据（飞书API不可用时的降级操作）；SLA计时规则：飞书API正常同步时以飞书申请提交时间（API返回字段）为起点，手动导入时以系统导入操作时间（管理员触发时刻）为起点；节假日照算自然日，截止时间为第3日23:59（UTC+8）
- **FR16：** 审核员可以在审核工作台列表页直接一键通过入群申请，系统自动调用飞书API批准入群
- **FR17：** 审核员可以拒绝入群申请并填写拒绝原因；拒绝原因为空时系统使用默认文案
- **FR18：** 系统可以向被拒绝用户提供包含拒绝原因和修改指引的H5页面
- **FR19：** 系统可以通过定时轮询、飞书webhook回调、管理员手动触发三种方式同步飞书入群状态
- **FR20：** 系统可以拦截审核进行中的重复提交；用户被拒后可重新提交，旧记录自动归档，SLA重置

### 消息通知

- **FR21：** 系统可以通过微信服务号向完成OpenID绑定的用户推送审核通过通知
- **FR22：** 系统可以通过微信服务号向用户推送审核拒绝通知（含拒绝原因和H5修改指引链接）
- **FR23：** 系统可以将推送失败的通知自动记录至失败队列
- **FR24：** 客服可以查看消息通知失败记录（通知时间、右豹编码、通知场景、渠道、推送状态）

### SLA 超时预警

- **FR25：** 系统可以自动识别超过3自然日（第3日23:59 UTC+8）未处理的待审核申请并向所有审核员发送预警通知。通知渠道：**微信服务号推送**（OpenID已绑定时首选）；**系统内顶部Banner通知**（兜底，所有有效审核员账号登录后均可见，直至申请处理完毕）。
- **FR26：** 系统可以在首次预警发出8小时后、申请仍未处理时向审核员发送二次催促通知，此后不再重复。通知渠道同 FR25。
- **FR27：** 管理员/审核员可以查看SLA预警记录（预警时间、申请记录ID、预警类型、发送状态）

### 飞书好友管理

- **FR28：** 操作人可以查看待入群/入群审核中用户的优先级列表。列表默认排序规则（三级）：**第一级**：入群状态（待入群排最前，即审核已通过但尚未建立好友关系；其次为审核中）；**第二级**：近10天动作数降序（动作越多越靠前，反映用户意向强度）；**第三级**：好友申请时效（有申请中记录的，发出时间越早越靠前以防超时；无申请记录的，按用户录入时间倒序）。列表顶部以分组分隔线区分"待入群（X人）"和"审核中（X人）"两个分组。好友申请被拒绝的记录自动移出主列表进入「需人工决策」分区，操作人可选择重新申请（含二次确认弹窗）或标记放弃。
- **FR29：** 客服/导师可以在系统内完成飞书账号OAuth授权，以支持以其身份发起好友申请
- **FR30：** 操作人可以以指定客服或导师的身份向用户发起飞书好友申请
- **FR31：** 系统可以通过飞书API回调自动更新好友申请状态（待接受/已接受/已拒绝/超时）
- **FR32：** 操作人可以手动标记"我已添加"以更新好友状态

### 组织管理

- **FR33：** 管理员可以新增、修改、删除客服记录，含上传企微二维码（jpg/png格式）
- **FR34：** 管理员可以新增、修改、删除导师记录；导师必须归属一个门派
- **FR35：** 管理员可以新增、修改、删除门派记录；门派侧不可操作导师归属
- **FR36：** 系统可以通过第三方API定时同步导师的学员绑定关系、项目数量和收益数据
- **FR37：** 系统可以在删除客服/导师/门派时检查关联数据，关联数>0时拦截并提示先解除关联
- **FR38：** 系统可以按组织层级（门派→导师→用户）聚合展示学员数量和收益数据，支持点击查看明细

### 数据看板

- **FR39：** 管理员可以查看全局运营看板（付费/普通用户数、各门派概览、各导师概览、各客服建联数）
- **FR40：** 客服可以查看个人建联数据看板（仅本人数据）
- **FR41：** 用户可以对看板数据按日期范围筛选（本月/上月/本季度/上季度/自定义区间，共5个快捷选项）
- **FR42：** 管理员可以查看录入→入群→付费转化漏斗数据（三步漏斗；各步骤以对应时间范围内新增量为统计口径；"咨询"步骤因系统无法结构化采集而去除）

### 权限与账号管理

- **FR43：** 管理员可以创建和配置角色组，为其分配菜单权限、操作权限和字段显隐权限
- **FR44：** 管理员可以创建后台账号，指定所属角色组和是否为审核员属性
- **FR45：** 账号持有人可以自助修改自己的昵称和登录密码

### 操作审计

- **FR47：** 管理员可以查看各业务表的操作日志记录（用户主档、付费记录、入群审核、客服、导师、门派）

### 历史数据迁移

- **FR48：** 管理员可以将存量Excel历史数据批量导入系统（用户主档、付费记录），系统执行字段映射和数据校验。**迁移策略框架**：①迁移范围覆盖用户主档（含归属关系）和付费记录两张表；②迁移入库时所有记录编码校验采用"跳过实时校验+批量补校验"策略（不走实时API，避免迁移过程因API抖动中断）——所有迁移记录强制标记为 `编码待验证`，迁移完成后系统后台批量触发补校验（≤100条/批、1秒间隔）；③补校验完成后生成校验结果报告（有效/无效/未响应各条数）供管理员人工处理无效记录；④完整字段映射规则、数据清洗逻辑及测试方案在**架构设计阶段**产出独立迁移规格附录文档。

## 非功能需求

### 性能

- **NFR1：** 常规列表页（用户主档、入群审核工作台）首屏加载时间 ≤ 3秒（标准网络条件，数据量≤10,000条）
- **NFR2：** 右豹编码实时校验（单条录入时的API响应）≤ 2秒，超时需提示用户等待而非静默失败
- **NFR3：** Excel批量导入（≤500条）从提交到结果展示 ≤ 30秒
- **NFR4：** 定时同步任务（右豹API、第三方API数据刷新）异步执行，不阻塞前端操作和用户交互
- **NFR5：** Dashboard数据看板在缓存数据有效时 ≤ 2秒加载完成

### 安全

- **NFR6：** 所有前后端通信强制使用HTTPS，不支持HTTP降级
- **NFR7：** 用户登录密码使用加盐哈希存储（不可逆），禁止明文存储
- **NFR8：** 微信OpenID仅存储于数据库，不在任何API响应或前端页面中暴露
- **NFR9：** 登录会话采用滑动过期策略（Sliding Session）：用户每次有效操作（任意 API 请求成功）自动重置过期计时器；**无操作超过8小时**后 Token 自动失效，需重新登录。Token 剩余15分钟时前端弹窗提示用户续期，避免操作中途被踢出。
- **NFR10：** 操作日志记录不可删除、不可篡改，作为完整审计链

### 可靠性与可用性

- **NFR11：** 系统核心功能（用户主档录入、入群审核工作台）在任意单一外部API不可用时可降级运行，不完全中断
- **NFR12：** SLA定时扫描任务执行失败时有自动重试机制（建议：失败后5分钟重试，最多3次）
- **NFR13：** 消息通知发送为异步操作，通知失败不影响审核操作本身的成功状态
- **NFR14：** 关键业务操作（审核通过/拒绝）需保证幂等性，防止网络重试导致重复执行
- **NFR15：** 系统工作时间（工作日08:00-22:00）核心功能可用性 ≥ 99%

### 数据完整性

- **NFR16：** 右豹编码唯一性约束在数据库层强制执行，应用层校验作为第一道防线，数据库唯一索引作为最终保障
- **NFR17：** 付费状态变更（已付费 ↔ 未付费）通过数据库事务保证原子性，不允许中间状态
- **NFR18：** 外部API缓存数据必须标注最后同步时间，页面区分展示"实时数据"与"缓存数据"
- **NFR19：** 历史数据迁移完成后，提供数据对比校验报告（导入条数 vs 原Excel条数，关键字段完整率）
- **NFR20：** 批量审核通过操作（含批量通过、批量拒绝）通过后端任务队列异步执行；飞书 API 调用速率不超过50次/秒（对齐飞书开放平台限额）；前端展示异步处理进度，不同步等待全部完成。单次批量操作上限统一为1000条，超出时提示分批处理。
- **NFR21：** 所有主列表页（用户主档、入群审核工作台、付费记录、通知失败记录、SLA预警记录、组织管理各列表）须实现分页，具体策略：主列表每页20条，入群审核工作台每页50条（高频操作场景），Dashboard概览表格不分页（全量展示，上限50条，超出提示跳转管理页），详情页操作日志默认展示最近20条并提供[加载更多]按钮。

## 补充决策记录

> 本节记录在原型设计阶段（prototype-spec.md v1.0~v1.4）经 Maziluo 确认、对 PRD 原始定义产生变更或补充的决策，供后续开发阶段参考。

### 功能需求变更

**FR42 转化漏斗步骤调整**
- **原定义：** 咨询 → 录入 → 入群 → 付费（四步）
- **变更后：** 录入 → 入群 → 付费（三步）
- **原因：** 系统无法结构化采集"咨询"数据，该步骤在后台无对应数据实体，去除以确保漏斗数据准确性
- **确认时间：** 2026-04-03

**FR15 SLA 计时规则补充**
- **原定义：** "SLA 以导入记录中的申请时间为计时起点"（仅针对手动导入场景）
- **补充说明：** 本文档明确区分两种场景：飞书 API 正常时以飞书申请提交时间为准；飞书 API 降级手动导入时以系统导入操作时间（管理员触发时刻）为准
- **确认时间：** 2026-04-03

**FR41 日期范围筛选快捷选项补充**
- **原定义：** 本月 / 本季度 / 自定义区间
- **补充后：** 本月 / 上月 / 本季度 / 上季度 / 自定义区间（共 5 项）
- **原因：** 管理层存在月度和季度环比分析需求
- **确认时间：** 2026-04-03

### 范围界定补充

**FR13 / FR18 H5 页面不在后台原型范围**
- 微信服务号 OpenID 绑定页（FR13）和被拒绝用户 H5 修改指引页（FR18）属于用户侧功能，不在后台管理系统原型设计范围内，由前端单独设计交付
- **确认时间：** 2026-04-03
