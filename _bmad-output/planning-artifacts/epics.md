# 客户服务中心 - Epic 需求分解

## 概述

本文档将客户服务中心的 PRD、UX 规范和架构设计需求分解为可实施的 Epic 和 Story，覆盖 FR1–FR48 全部功能需求及对应的技术架构要求。

---

## 需求清单

### 功能需求（FR）

**用户数据管理**

- FR1: 客服可以录入用户主档信息（右豹编码、右豹ID、飞书ID、飞书手机号、飞书昵称）
- FR2: 系统可以在录入时通过右豹APP API实时校验右豹编码的真实性，校验失败时阻断提交。API不可达时，普通客服无法绕过，提示等待重试；管理员可见「暂跳过校验」按钮，跳过记录标记为 `编码待验证`，API恢复后30分钟内自动触发补校验
- FR3: 系统可以自动校验飞书手机号格式，格式不合规时提示错误
- FR4: 客服可以查看用户主档列表，包含聚合展示的所属客服、所属导师、所属门派信息
- FR5: 客服可以修改用户主档中的可编辑字段
- FR6: 管理员可以删除用户主档记录（关联数据>0时系统拦截并提示，关联数据包含入群审核记录和付费记录）
- FR7: 负责人可以单条手动新增付费记录
- FR8: 负责人可以通过Excel批量导入付费记录，系统识别并报告失败条目，支持导出失败明细
- FR9: 系统可以在付费录入时校验合法性：右豹编码须存在、不可重复录入已付费用户、付费金额不可为负数
- FR10: 负责人可以修改付费记录的付费金额、付费时间、付费对接人字段
- FR11: 负责人可以删除付费记录，删除后对应用户"是否付费学员"状态自动回退为未付费（数据库事务保证原子性）
- FR12: 系统可以通过右豹APP API定时同步用户的右豹总关键词数、总订单数、项目收益数据（每小时+支持手动触发）
- FR13: 用户可以通过微信服务号H5菜单完成OpenID与右豹编码的绑定（**不在后台管理系统范围，由前端单独设计**）

**入群审核**

- FR14: 系统可以通过飞书API自动获取飞书群入群申请并生成审核记录（5分钟轮询+Webhook回调）
- FR15: 管理员可以手动导入入群申请数据（飞书API不可用时的降级操作）；SLA计时：飞书API正常时以飞书申请提交时间为起点，手动导入时以系统导入操作时间为起点；截止时间为第3日23:59（UTC+8）
- FR16: 审核员可以在审核工作台列表页直接一键通过入群申请，系统自动调用飞书API批准入群
- FR17: 审核员可以拒绝入群申请并填写拒绝原因；拒绝原因为空时系统使用默认文案
- FR18: 系统可以向被拒绝用户提供包含拒绝原因和修改指引的H5页面（**不在后台管理系统范围**）
- FR19: 系统可以通过定时轮询、飞书webhook回调、管理员手动触发三种方式同步飞书入群状态；Webhook超时30分钟内未回调时系统自动触发补偿性轮询
- FR20: 系统可以拦截审核进行中的重复提交（飞书API端静默丢弃）；用户被拒后可重新提交，旧记录自动归档，SLA重置；归档规则：重新提交立即归档、超30日系统自动归档、管理员手动归档

**消息通知**

- FR21: 系统可以通过微信服务号向完成OpenID绑定的用户推送审核通过通知
- FR22: 系统可以通过微信服务号向用户推送审核拒绝通知（含拒绝原因和H5修改指引链接）
- FR23: 系统可以将推送失败的通知自动记录至失败队列
- FR24: 客服可以查看消息通知失败记录（通知时间、右豹编码、通知场景、渠道、推送状态）

**SLA 超时预警**

- FR25: 系统可以自动识别超过3自然日（第3日23:59 UTC+8）未处理的待审核申请并向所有审核员发送预警通知；通知渠道：微信服务号推送（OpenID已绑定时首选）+ 系统内顶部Banner兜底（持续至申请处理完毕，所有有效审核员账号可见）
- FR26: 系统可以在首次预警发出8小时后、申请仍未处理时向审核员发送二次催促通知，此后不再重复；通知渠道同FR25
- FR27: 管理员/审核员可以查看SLA预警记录（预警时间、申请记录ID、预警类型、发送状态）

**飞书好友管理**

- FR28: 操作人可以查看待入群/入群审核中用户的优先级列表；三级排序：①待入群>审核中；②近10天动作数降序；③好友申请时效越早越靠前；列表顶部分组分隔线区分"待入群（X人）"和"审核中（X人）"；被拒绝记录移入「需人工决策」分区（可重新申请含二次确认弹窗，或标记放弃）；好友申请7自然日无响应自动标记超时，主视图默认隐藏超时记录
- FR29: 客服/导师可以在系统内完成飞书账号OAuth授权，以支持以其身份发起好友申请；OAuth回调状态跨标签页同步使用 BroadcastChannel API
- FR30: 操作人可以以指定客服或导师的身份向用户发起飞书好友申请
- FR31: 系统可以通过飞书API回调自动更新好友申请状态（待接受/已接受/已拒绝/超时）
- FR32: 操作人可以手动标记"我已添加"以更新好友状态

**组织管理**

- FR33: 管理员可以新增、修改、删除客服记录，含上传企微二维码（jpg/png格式）
- FR34: 管理员可以新增、修改、删除导师记录；导师必须归属一个门派
- FR35: 管理员可以新增、修改、删除门派记录；门派侧不可操作导师归属
- FR36: 系统可以通过第三方API定时同步导师的学员绑定关系、项目数量和收益数据（每小时，Redis缓存，支持手动触发）
- FR37: 系统可以在删除客服/导师/门派时检查关联数据，关联数>0时拦截并提示先解除关联
- FR38: 系统可以按组织层级（门派→导师→用户）聚合展示学员数量和收益数据，支持点击查看明细（门派详情页新增「旗下导师」Tab，导师详情页新增「名下学员」Tab）

**数据看板**

- FR39: 管理员可以查看全局运营看板（付费/普通用户数、各门派概览、各导师概览、各客服建联数）
- FR40: 客服可以查看个人建联数据看板（仅本人数据）
- FR41: 用户可以对看板数据按日期范围筛选（本月/上月/本季度/上季度/自定义区间，共5个快捷选项）
- FR42: 管理员可以查看录入→入群→付费转化漏斗数据（三步漏斗；各步骤以对应时间范围内新增量为统计口径）

**权限与账号管理**

- FR43: 管理员可以创建和配置角色组，为其分配菜单权限、操作权限和字段显隐权限
- FR44: 管理员可以创建后台账号，指定所属角色组和是否为审核员属性；账号必须先禁用才可删除；历史操作日志随账号删除后保留不删
- FR45: 账号持有人可以自助修改自己的昵称和登录密码（通过顶部导航栏下拉菜单进入个人设置页）

**操作审计**

- FR46: 系统可以为所有数据变更操作（录入/修改/删除/审核）自动记录操作日志，含操作人、操作时间、操作前后数据快照；通过NestJS Interceptor自动记录，不依赖手动调用
- FR47: 管理员可以查看各业务表的操作日志记录（用户主档、付费记录、入群审核、客服、导师、门派）

**历史数据迁移**

- FR48: 管理员可以将存量Excel历史数据批量导入系统（用户主档、付费记录）；迁移策略：跳过实时API校验，所有记录强制标记为 `编码待验证`，迁移完成后后台批量触发补校验（≤100条/批、1秒间隔），生成校验结果报告（有效/无效/未响应各条数）；导出功能：用户主档、付费记录、入群审核三模块支持按当前筛选结果导出Excel，导出文件链接保留24小时

### 非功能需求（NFR）

**性能**

- NFR1: 常规列表页（用户主档、入群审核工作台）首屏加载时间 ≤ 3秒（标准网络条件，数据量≤10,000条）
- NFR2: 右豹编码实时校验（单条录入时的API响应）≤ 2秒，超时需提示用户等待而非静默失败
- NFR3: Excel批量导入（≤500条）从提交到结果展示 ≤ 30秒
- NFR4: 定时同步任务（右豹API、第三方API数据刷新）异步执行，不阻塞前端操作和用户交互
- NFR5: Dashboard数据看板在缓存数据有效时 ≤ 2秒加载完成

**安全**

- NFR6: 所有前后端通信强制使用HTTPS，不支持HTTP降级
- NFR7: 用户登录密码使用bcrypt加盐哈希存储（不可逆），禁止明文存储
- NFR8: 微信OpenID仅存储于数据库，不在任何API响应或前端页面中暴露
- NFR9: 登录会话采用滑动过期策略：用户每次有效操作自动重置计时器；无操作超过8小时后Token自动失效；Token剩余15分钟时前端弹窗提示续期
- NFR10: 操作日志记录不可删除、不可篡改（数据库应用账号仅有INSERT/SELECT权限）

**可靠性与可用性**

- NFR11: 系统核心功能（用户主档录入、入群审核工作台）在任意单一外部API不可用时可降级运行，不完全中断
- NFR12: SLA定时扫描任务执行失败时有自动重试机制（失败后5分钟重试，最多3次）
- NFR13: 消息通知发送为异步操作，通知失败不影响审核操作本身的成功状态
- NFR14: 关键业务操作（审核通过/拒绝）需保证幂等性，防止网络重试导致重复执行
- NFR15: 系统工作时间（工作日08:00-22:00）核心功能可用性 ≥ 99%

**数据完整性**

- NFR16: 右豹编码唯一性约束在数据库层强制执行（唯一索引），应用层校验作为第一道防线
- NFR17: 付费状态变更（已付费 ↔ 未付费）通过数据库事务保证原子性，不允许中间状态
- NFR18: 外部API缓存数据必须标注最后同步时间，页面区分展示"实时数据"与"缓存数据"
- NFR19: 历史数据迁移完成后，提供数据对比校验报告（导入条数 vs 原Excel条数，关键字段完整率）
- NFR20: 批量审核通过操作通过后端任务队列异步执行；飞书API调用速率不超过50次/秒；前端展示异步处理进度；单次批量操作上限统一为1000条
- NFR21: 所有主列表页须实现分页：主列表每页20条，入群审核工作台每页50条，Dashboard概览不分页（上限50条，超出提示跳转管理页），详情页操作日志默认最近20条+[加载更多]按钮

### 架构衍生需求（来自 Architecture.md）

**项目初始化**

- ARCH1: Monorepo 结构（pnpm workspace）：`kefu-backend`（NestJS 10.x + TypeScript 5.x）+ `kefu-frontend`（Vue3 + Vite 5.x）；使用 `npx @nestjs/cli new` 和 `pnpm create vite@latest` 初始化
- ARCH2: Docker Compose 部署拓扑：nginx（HTTPS终止+静态资源）、backend（NestJS，端口3000）、mysql（8.0，仅内网可访问）、redis（7.x，仅内网可访问）、worker（BullMQ独立进程）
- ARCH3: 环境配置：`.env.example` 提交到Git（含所有变量名），`.env.local`（开发）和 `.env.production`（生产）不提交；覆盖 DATABASE_URL、REDIS_URL、JWT_SECRET、JWT_EXPIRES_IN、4路API连接变量

**数据库**

- ARCH4: Prisma ORM with MySQL 8.0；BIGINT AUTO_INCREMENT 主键；snake_case 复数表名；`prisma migrate dev`（开发）/ `prisma migrate deploy`（生产）
- ARCH5: 审计日志表（audit_logs）的数据库应用账号仅有INSERT/SELECT权限（无DELETE/UPDATE），实现不可篡改
- ARCH6: 右豹编码唯一索引 `idx_users_right_leopard_code`；外键在应用层校验（数据库层不设置级联删除）

**后端架构**

- ARCH7: BullMQ 队列处理：通知发送队列、SLA扫描队列、批量审核队列、API同步队列
- ARCH8: @nestjs/schedule 定时任务：右豹API每小时同步、飞书好友状态每日刷新、第三方导师API每小时同步、飞书入群申请5分钟轮询
- ARCH9: 统一API响应格式 `{code: 0, data: T, message: 'success'}` + 业务错误码表（10001–50003）
- ARCH10: JWT Passport 认证；权限payload内嵌JWT（减少DB查询）；RBAC Guard 在 Controller 层拦截

**前端架构**

- ARCH11: Pinia Store 模块：auth（账号+权限+登录状态）、user、audit、dashboard、notification、app（全局API降级状态）
- ARCH12: 定制业务组件：CodeVerifyInput（三态）、MetricCard（玻璃拟态卡片）、SlaStatusBadge（自动计算超时）、ApiStatusBar（正常时不渲染）、ConversionFunnel
- ARCH13: 路由权限守卫：无权限路由 redirect 404（非403，避免信息泄露）；Vue composable `usePermission()` 统一权限判断
- ARCH14: OAuth回调状态跨标签页同步使用 BroadcastChannel API（非localStorage事件）

**缓存策略**

- ARCH15: Redis 缓存 Key/TTL：用户动作数据 `user:actions:{code}` 3600s、导师统计 `mentor:stats:{mentorId}` 3600s、全局Dashboard `dashboard:global:{dateRange}` 300s、个人Dashboard `dashboard:agent:{agentId}:{dateRange}` 300s、飞书好友状态 `lark:friend:{userId}` 86400s

### UX 设计需求（来自 UX Design Specification）

**视觉设计系统**

- UX-DR1: 深色玻璃拟态风格：主背景 `#0f172a`（slate-950）、次背景 `#1e293b`（slate-800）、强调色 `#6366f1`（indigo-500）、玻璃层 `rgba(255,255,255,0.08)` + `backdrop-blur-xl`、玻璃边框 `rgba(255,255,255,0.12)`
- UX-DR2: 状态色语义系统：红 `#ef4444`（超时/错误）、橙 `#f97316`（待处理/警告）、绿 `#22c55e`（通过/成功）、灰 `#475569`（已归档/禁用）、蓝 `#38bdf8`（系统信息/API降级）；状态标识同时附有文字标签和图标（色觉障碍友好）
- UX-DR3: 排版：Inter/PingFang SC 组合；JetBrains Mono 用于编码等宽字段；Dashboard 核心指标 32px/700weight；正文 14px；辅助说明 12px
- UX-DR4: 布局：左侧导航栏 240px 固定 + 内容区最大 1200px；页面内边距 24px；卡片间距 16px；圆角：卡片 12px、按钮 8px、标签 6px

**导航与布局**

- UX-DR5: 左侧固定导航 + 角色登录后菜单项按权限自动裁剪（v-if 完全不渲染，非 v-show 置灰）；无权限菜单对用户完全不可见
- UX-DR6: 右侧抽屉（Drawer）模式用于新建/编辑表单，不离开列表页上下文
- UX-DR7: 列表页行内操作——高频操作（审核通过/拒绝按钮）直接暴露在列表行，无需跳转详情页

**定制业务组件**

- UX-DR8: CodeVerifyInput 组件三态：loading（输入框右侧转圈）/ valid（绿色边框+✓+"编码有效"）/ invalid（红色边框+✗+具体错误说明），失焦后自动触发API校验
- UX-DR9: MetricCard 玻璃拟态指标卡片：`bg-white/10 backdrop-blur-md border border-white/20 shadow-xl` 基础样式，用于 Dashboard 数据展示
- UX-DR10: SlaStatusBadge 状态徽标自动计算超时状态，红色（已超时）/橙色（待处理），附剩余时间/已超时时长展示；SLA超时记录行整行红色高亮
- UX-DR11: ApiStatusBar API降级状态条，正常时完全不渲染，降级时展示蓝色 `#38bdf8` 信息条（非侵入性，非弹窗）
- UX-DR12: ConversionFunnel 转化漏斗图（三步：录入→入群→付费），参考 Grafana/DataDog 数据密度风格

**交互模式**

- UX-DR13: Toast 反馈用于非高风险操作（审核通过、状态更新）；二次确认弹窗仅用于高风险操作（删除含关联数据记录、重新发起被拒绝好友申请等）；删除前展示影响范围
- UX-DR14: Token 剩余 15 分钟时前端弹窗提示用户续期，避免操作中途被踢出（对应 NFR9 Sliding Session）
- UX-DR15: 新增用户 → 右侧抽屉滑入 → 编码校验 → 字段填写 → 保存成功 → 抽屉关闭 + 新条目高亮 + Toast "用户已录入"（定义性交互流程）
- UX-DR16: 无障碍：正文深色背景对比度 ≥ 4.5:1（WCAG AA）；所有交互元素保留可见焦点环 `ring-2 ring-indigo-500`

### Story 9.1：Dashboard 数据聚合接口与 Redis 缓存层

作为**系统（后端服务）**，
我希望 Dashboard 所有聚合数据通过 Redis 缓存提供，写操作后主动失效，
以便 Dashboard 在缓存有效时 ≤ 2 秒加载完成（满足 NFR5、ARCH15）。

**验收标准：**

**Given** 管理员请求全局 Dashboard 数据
**When** Redis 缓存命中（`dashboard:global:{dateRange}`，TTL 300s）
**Then** 直接返回缓存数据，响应时间 ≤ 2 秒（满足 NFR5）

**Given** Redis 缓存未命中（首次加载或缓存过期）
**When** 数据库查询聚合完成
**Then** 结果写入 Redis 缓存（TTL 300s），同时返回给前端

**Given** 用户主档、付费记录、入群审核等业务表发生写操作
**When** 写操作完成
**Then** 相关 Dashboard 缓存主动失效（`del dashboard:global:*`），确保下次读取获取最新数据

**Given** 客服个人 Dashboard 请求
**When** Redis 缓存命中（`dashboard:agent:{agentId}:{dateRange}`，TTL 300s）
**Then** 返回该客服专属数据，不混入其他客服数据

**And** Dashboard 聚合接口支持 5 种日期范围参数：`current_month`、`last_month`、`current_quarter`、`last_quarter`、`custom`（满足 FR41）

## Epic 2：认证安全 · 账号权限体系

**目标：** 管理员可以安全登录系统、配置角色权限体系、创建团队账号，账号持有人可自助管理个人信息，管理员可查看操作日志，团队全员可控、可追溯地开始使用系统。

### Story 2.1：账号登录认证（JWT + Sliding Session）

作为**系统账号持有人**，
我希望通过账号名和密码安全登录系统，会话在活跃期间保持有效，长时间不操作自动退出，
以便系统操作安全可控，防止未授权访问。

**验收标准：**

**Given** 已创建的账号（loginId + 密码）
**When** POST `/api/v1/auth/login` 提交正确凭据
**Then** 返回 `{ accessToken: "...", expiresIn: 28800 }`，Token payload 包含 `sub`（accountId）、`roleId`、`isAuditor`、完整权限对象

**Given** 错误的账号名或密码
**When** POST `/api/v1/auth/login`
**Then** 返回 HTTP 401，`code: 50001`，不泄露是账号不存在还是密码错误（统一返回"账号或密码错误"）

**Given** 已登录用户持续进行 API 操作
**When** 每次有效 API 请求成功
**Then** Token 过期时间自动重置为从当前时刻起 8 小时（Sliding Session，满足 NFR9）

**Given** 已登录用户连续 8 小时无任何操作
**When** 尝试发起任意 API 请求
**Then** 返回 HTTP 401，`code: 50002`，前端跳转登录页

**Given** Token 剩余有效期 ≤ 15 分钟
**When** 前端检测到剩余时间
**Then** 弹出续期提示弹窗，告知用户点击任意操作即可自动续期，避免操作中途被踢出

**And** 密码以 bcrypt 加盐哈希方式存储，禁止任何接口以明文返回密码（满足 NFR7）

### Story 1.5：操作日志审计拦截器

作为**系统管理员**，
我希望所有数据变更操作被自动记录到审计日志，无需开发者手动调用，
以便任何数据变更都可追溯，满足内部审计需求（NFR10）。

**验收标准：**

**Given** `AuditLogInterceptor` 已全局注册到 NestJS 应用
**When** 任意 POST/PATCH/DELETE API 调用成功完成
**Then** `audit_logs` 表中自动新增一条记录，包含：`table_name`（当前操作的业务表名）、`record_id`（被操作记录的 ID）、`operator_id`（从 JWT Token `sub` 字段提取）、`action_type`（CREATE/UPDATE/DELETE）、`before_data`（操作前完整 JSON 快照）、`after_data`（操作后完整 JSON 快照）、`created_at`

**Given** 任意 GET 请求
**When** 拦截器执行
**Then** 不创建任何 `audit_logs` 记录

**Given** 审计日志写入失败（如数据库临时不可用）
**When** 主业务操作已成功完成
**Then** 主操作响应正常返回，不因审计失败而回滚主业务事务；审计失败写入应用错误日志供排查

**Given** 任何角色的任何账号
**When** 尝试通过系统 API 删除任意 `audit_logs` 记录
**Then** 系统不提供任何删除审计日志的 API 端点（API 层无 DELETE /audit-logs/:id 接口）；即使直接操作数据库，应用账号权限亦拒绝 DELETE（双重保障，满足 NFR10）
